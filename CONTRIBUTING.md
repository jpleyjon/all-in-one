# Contributing

Thank you for your interest in contributing to this project.

## Ways to contribute

- Report bugs and suggest enhancements through GitHub Issues.
- Submit code changes via pull requests.
- Improve documentation, examples, and project clarity.
- Review pull requests and test proposed changes.

## Before you start

- Read this document and the project's `CODE_OF_CONDUCT.md`.
- Search existing issues and pull requests before opening duplicates.
- If your change is large, open an issue first to align on scope.

## Reporting issues

When reporting a bug, include:

- A clear summary of what happened.
- Steps to reproduce.
- Expected behavior.
- Actual behavior.
- Version and environment details (Node.js version, OS, etc.).

For feature requests, include:

- The problem you want to solve.
- Why the change is needed.
- A rough idea of how it could be implemented.

## Making changes

- Keep pull requests focused and small when possible.
- Add or update tests where applicable.
- Match existing coding style and project conventions.
- Run tests before submitting, and include results in your PR description.

## Pull request checklist

- Link to a related issue, if one exists.
- Summarize what changed and why.
- Describe how you validated the change.
- Ensure CI (if enabled) is passing.
- Avoid unrelated refactors in the same PR.

## Commit style

Use clear, descriptive commit messages. For example:

- `feat: add input validation`
- `fix: handle empty config in parser`
- `docs: clarify API usage`

## Review process

Project maintainers will review and may ask for small follow-up changes.
Please be responsive to feedback so changes can be merged quickly.

## Code of conduct

All contributions must follow the project’s [Code of Conduct](CODE_OF_CONDUCT.md).
